# 测试计划与用例

## 1. 目的
明确测试范围、测试层级及核心业务场景的测试用例，确保“元数据驱动”架构下的数据一致性、问卷状态流转的正确性与系统稳定性。

## 2. 测试策略
- **单元测试 (Unit Test)**：利用 `pytest` 测试后端的 JSON 解析逻辑、问卷状态流转校验函数等核心算法。
- **集成测试 (Integration Test)**：测试 FastAPI 路由层与 SQLite 数据库的联调，重点验证 JSON 数据的完整存取及接口的权限控制。
- **端到端测试 (E2E Test)**：模拟教导主任创建问卷 -> 下发问卷 -> 教师填报 -> 主任查看数据并结束问卷的全链路。

## 3. 核心测试用例

### 3.1 动态增删列问卷编辑测试
- **用例标识**: TC-SURVEY-01
- **前置条件**: 以 Admin 身份登录，新建处于草稿状态的问卷。
- **操作步骤**: 
  1. 创建包含“文本”、“数字”两种基础字段的问卷并保存草稿。
  2. 重新编辑该问卷，增加一个“表格(Table)”复合字段并保存。
- **预期结果**: API 返回 `200 OK`，数据库 `survey_schema` 更新成功，前端能立即根据新 Schema 渲染出表格输入框。

### 3.2 问卷状态流转防呆测试
- **用例标识**: TC-SURVEY-02
- **前置条件**: 存在一个已下发 (published) 的问卷。
- **操作步骤**: 
  1. Admin 尝试调用 `PUT /api/surveys/{id}` 修改该问卷的 schema。
  2. Admin 尝试调用 `POST /api/surveys/{id}/publish` 再次下发。
- **预期结果**: 后端拦截请求，返回 `400 Bad Request`，提示“非草稿状态无法编辑”和“问卷已处于下发状态”。

### 3.3 数据填报与防篡改测试
- **用例标识**: TC-FILL-01
- **前置条件**: 发布一个有效问卷，以 User（教师）身份登录。
- **操作步骤**:
  1. 教师正常提交答卷数据。
  2. 教师尝试再次向该问卷发送 POST 提交请求。
- **预期结果**: 第一次提交成功；第二次提交后端校验拒绝，返回提示“您已提交过该问卷，请勿重复提交”，防止数据刷单或错乱。

### 3.4 问卷撤回/结束控制测试
- **用例标识**: TC-CONTROL-01
- **前置条件**: 发布一个有效问卷，且已有教师提交了答卷。
- **操作步骤**:
  1. Admin 在列表点击“截止问卷”，调用 `/close` 接口。
  2. 另一位未填报的教师尝试打开该问卷并提交。
- **预期结果**: Admin 成功截止问卷，状态变为 `closed`；未填报教师在打开问卷时或提交时，后端返回 `403 Forbidden`，提示“问卷已结束收集”。

### 3.5 复杂 JSON 数据查询与合并测试
- **用例标识**: TC-QUERY-01
- **操作步骤**: Admin 查询一份含有多个答卷记录的问卷，检查返回的列表接口。
- **预期结果**: 接口返回的数组中，每个对象的 `dynamic_data` 应当被正确展开为 JSON 对象（而非字符串），且能与静态字段平级合并返回给前端，供前端 Table 组件直接绑定渲染。

### 3.6 全员实时看板测试
- **用例标识**: TC-VIEW-01
- **前置条件**: 问卷已发布，教师 A 和 教师 B 同时在线。
- **操作步骤**:
  1. 教师 A 在看板页面建立 WebSocket 连接。
  2. 教师 B 提交一份答卷。
- **预期结果**: 教师 A 的看板无需手动刷新，WebSocket 接收到 `new_response` 事件，界面自动追加一条教师 B 的填报记录。
